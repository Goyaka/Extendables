factory¶

Error.factory(name)¶
Arguments:
  • name (String) – Preferably the same name as the variable you’re associating the error with.

Use this classmethod to make sure your custom error types work just like the built-in ones.

var DatabaseError = Error.factory("DatabaseError");

is¶

Error#is(type)¶
Arguments:
  • type (Constructor) – Use the constructor itself, not a string or an instance.
Returns Bool:

True or false. Any error type matches the base class, so new SyntaxError().is(Error) would return true.

This method overloads Object#is() to combat a problem with some versions of ExtendScript that leads to all error types being considered the base class Error. This problem makes it impossible to do simple comparisons on errors, for example new EvalError() instanceof SyntaxError. The previous expression should return false but will return true.

When testing whether you’re dealing with a specific kind of error, use this method, and refrain from using instanceof.

See also

error-handling has some useful advice on how to handle errors in ExtendScript.

try {
    raise new SyntaxError();
} catch (error if error.is(TypeError)) {
    alert("This displays in case of a type error, but not in case of a syntax error.");
}

class EnvironmentError()¶

class EvalError()¶

class ImportError()¶

Use when an IO operation (loading a file, writing to a file, an internet connection) fails.

class IOError()¶

Use to warn people that a feature has not yet been implemented, as a placeholder to remind yourself or to indicate that a subclass needs to overload the parent method.

class NotImplementedError()¶

class ParseError()¶
